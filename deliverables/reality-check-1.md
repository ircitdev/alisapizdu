## 3. Технические риски

### 3.1. SQLite при вирусном трафике

**Уровень риска: ВЫСОКИЙ**

**Заявка**: "SQLite WAL для конкурентного чтения"

**Реальность**:
- SQLite WAL разрешает множественное чтение + одну запись одновременно
- При 100к пользователей в день, 5 нажатий/юзер = 500к записей/день
- Каждый запрос = 1 INSERT (сообщение) + 1 UPDATE (ответ AI) + SSE broadcast
- `busy_timeout = 5000` -- при высокой нагрузке пользователи будут ждать 5 секунд на каждую запись
- SQLite -- single-writer. При 50+ одновременных записях в секунду -- bottleneck

**Реальный предел**: SQLite WAL на хорошем SSD -- ~1000-3000 записей/сек. При пиковой нагрузке (50+ rps на запись) будет работать, но с задержками. При 500+ rps -- деградация.

**Митигация**: Для MVP -- допустимо. Но нужен план миграции на PostgreSQL (Drizzle/Prisma позволяют переключить). Write batching или очередь записей через in-memory queue.

### 3.2. SSE масштабирование

**Уровень риска: ВЫСОКИЙ**

**Заявка**: "In-memory pub/sub, SSE broadcast"

**Реальность**:
- Каждое SSE соединение = открытое TCP соединение
- Node.js single process: реалистичный лимит ~5000-10000 одновременных SSE соединений (зависит от RAM)
- При 10к одновременных пользователей с SSE = 10к открытых соединений = ~2-4GB RAM только на соединения
- broadcast на 10к клиентов при каждом новом сообщении = CPU spike
- nginx default: `worker_connections 1024` -- нужно явно увеличивать
- nginx proxy_read_timeout для SSE нужно ставить высоким (или отключать)

**Митигация для MVP**:
- Ограничить SSE соединения до 5000
- При превышении -- фоллбэк на polling (каждые 3 сек)
- PM2 cluster mode НЕ работает с in-memory SSE (разные процессы не шарят Map)
- Решение: Redis pub/sub или sticky sessions

**Критическая проблема**: Архитектура с in-memory SSE manager НЕСОВМЕСТИМА с PM2 cluster mode. Это означает: один процесс Node.js, один CPU core. При вирусном трафике -- bottleneck.

### 3.3. OpenAI API costs

**Уровень риска: СРЕДНИЙ**

**Расчет при вирусном трафике**:
- gpt-4o-mini: ~$0.15 / 1M input tokens, ~$0.60 / 1M output tokens
- Системный промпт ~200 tokens + вопрос ~20 tokens = ~220 input tokens/запрос
- Ответ ~100 tokens (ограничение max_tokens: 200)
- При 100к запросов/день:
  - Input: 100k * 220 = 22M tokens = ~$3.30
  - Output: 100k * 100 = 10M tokens = ~$6.00
  - **Итого: ~$9.30/день = ~$280/месяц**
- При 1М запросов/день (если РЕАЛЬНО завирусится):
  - **~$93/день = ~$2800/месяц**

**Митигация**: Mock responses для 80% трафика, OpenAI для 20% (рандомный отбор или каждый N-й запрос). Это снизит расходы в 5 раз. Кэширование ответов -- нет смысла (все вопросы одинаковые, но ответы должны быть разные).

## 5. Нагрузка: что если придет 100к человек за день

### Сценарий: пост в Telegram-канале или TikTok, 100к уникальных за 24ч

**Профиль нагрузки**:
- Пик: 20% трафика за 2 часа = 20к уникальных/2ч = ~3 rps страниц
- Каждый пользователь: 1 загрузка + 3-5 нажатий кнопки = ~5 API запросов
- Пиковая нагрузка: ~15-25 rps на API + 3000-5000 SSE соединений

**Bottlenecks по приоритету**:

1. **SSE broadcast** -- 5000 соединений, каждое новое сообщение шлется всем = 5000 * ~200 bytes = 1MB на каждое сообщение. При 15 rps = 15MB/сек outbound. Нужен хороший канал.

2. **SQLite write lock** -- 15 записей/сек (INSERT + UPDATE) -- SQLite справится. Но concurrent reads при записи могут замедляться.

3. **Node.js event loop** -- SSE broadcast + AI запросы + DB queries. На одном процессе при 5000 SSE клиентов event loop будет загружен. Latency ответов вырастет.

4. **nginx** -- нужно настроить:
   ```
   worker_connections 16384;
   proxy_read_timeout 86400;  # для SSE
   proxy_buffering off;       # для SSE
   ```

5. **OpenAI rate limits** -- gpt-4o-mini: 500 RPM на free tier, 10000 RPM на paid. При 15 rps = 900 RPM. Нужен paid tier.

**Рекомендация**: VPS минимум 4GB RAM, 2 vCPU, 100Mbps. Стоимость: ~1000-2000 руб/мес (Hetzner, Timeweb, Selectel).

## 6. Контент-риски

### 6.1. Модерация платных сообщений

**Что будут писать за 1000р** (реалистичный прогноз):
- 40% -- шутки и мемы (безобидные)
- 30% -- мат и провокации (но не криминальные)
- 15% -- реклама и спам ("Подписывайтесь на мой канал")

- 5% -- персональные данные, доксинг

**Проблема**: 10% токсичного контента + 15% спама -- это каждый 4-й платный пост.

**Решение для MVP (без автомодерации)**:
- Pre-moderation: платные сообщения публикуются ПОСЛЕ ручной проверки
- Telegram-бот для модератора: новый платный пост -> уведомление -> кнопки "Одобрить / Отклонить"
- Blacklist слов: базовый regex на экстремизм, угрозы, персданные
- Refund policy: если сообщение отклонено модератором -- возврат денег

### 6.2. Бесплатные сообщения

Все бесплатные сообщения одинаковые ("Алиса покажи пизду") -- модерация не нужна. Это плюс архитектуры.
